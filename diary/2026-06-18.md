## What Happened

### Portable CUDA build
The previous session had hardcoded `/usr/lib/x86_64-linux-gnu` in the CGO LDFLAGS in `cuda_backend.go`. Removed it — now just `-lcudart` with no explicit `-L` path, so the linker uses whatever `LD_LIBRARY_PATH` provides. This makes the build portable across CUDA installations. Wrote a `Makefile` with two targets: `libcuda_render.so` (sm_75..sm_90, CUDA 12.x) and `libcuda_render-cuda13.so` (adds sm_100/sm_120, CUDA 13+).

Both tests now pass.

### TessellatedObjColl seam investigation
The session note "zero-density gaps at unit-cell boundaries" was a theoretical risk with no reproduction. Traced through the code carefully:
- The folding formula `x = x - dx*floor((x-UC.Xmin)/dx)` always produces values in `[UC.Xmin, UC.Xmax)`.
- `UnitCell.Density` uses strict `>` on the upper bound, so `x = UC.Xmax` (which can't be produced by folding) is a non-issue.
- Verified analytically for Kelvin struts: e.g. the z-face pair `(0.5,0,0.75)→(0.5,0.25,1.0)` and `(0.5,0,0.25)→(0.5,0.25,0.0)` give identical density at symmetric points across z=1.

**Conclusion: no code bug.** Wrote three regression tests to pin this correct behavior.

### AffineDeformation documentation
`AffineDeformation.Apply` returns `M*v` (pure linear map), while `GaussianDeformation.Apply` returns `v + displacement(v)`. This semantic difference was undocumented and could silently mislead users setting matrix entries expecting displacement behavior. Added a comment and wrote tests pinning both semantics explicitly.
